## Context

目前會議呈現頁的簽到彈窗把一般簽到、代簽、更正、補簽共用在同一組表單欄位中，導致 `一般簽到` 也必須手動填寫部門與姓名。這個做法符合前一版前端原型的手動簽到假設，但與實際落地時會由登入系統提供使用者唯一識別、姓名與部門的前提不一致。由於簽到紀錄仍需保留 `memberId`、`departmentId`、`memberName` 等可追溯欄位，這次調整不是單純刪掉兩個欄位，而是要把身份來源從表單輸入改成登入上下文，同時維持管理型流程的稽核能力。

## Goals / Non-Goals

**Goals:**

- 讓一般簽到改為系統自動帶入身份，縮短自助簽到流程並減少誤填。
- 讓代簽、更正、補簽維持清楚分流，不因一般簽到簡化而失去目標選擇與原因欄位。
- 明確定義抓不到登入身份時的阻擋規則，避免退回自由輸入破壞資料一致性。
- 保持 attendance ledger 的可追溯欄位完整，讓後續報表、重複簽到判斷與更正流程不被破壞。

**Non-Goals:**

- 這次不導入真正的後端登入整合或新認證依賴，只先在規格與任務層定義身份由系統上下文提供。
- 這次不重做整個會議簽到資訊架構，只聚焦一般簽到與管理型流程的欄位分流。
- 這次不改變 append-only ledger、代簽稽核、補簽逾期與更正鏈的既有治理原則。

## Decisions

### Decision: Derive self sign-in identity from authenticated meeting context

一般簽到改為直接使用系統提供的登入者 `memberId`、`memberName`、`departmentId`，彈窗只顯示唯讀身份確認，不再讓使用者手填姓名與部門。這樣能讓自助簽到流程與真實落地環境一致，並減少因手動輸入造成的錯誤比對。

替代方案是保留欄位但先自動帶預設值；但此作法仍允許使用者改壞資料，無法真正落實「身份由系統決定」的邊界，因此不採用。

### Decision: Keep management sign-in modes explicitly target-driven

代簽、更正、補簽仍保留各自必要的目標選擇與原因欄位，不與一般簽到共用同一組最小化欄位。這樣可維持管理型操作的稽核語意，避免因為一般簽到簡化而讓「替誰代簽」「更正哪筆紀錄」變得不明確。

替代方案是把所有模式都做成同一個簡化彈窗；但這會模糊一般簽到與管理型操作的責任邊界，也不利於後續權限治理，因此不採用。

### Decision: Block self sign-in when identity context is incomplete

若系統在簽到當下無法提供唯一身份、姓名或部門，系統應直接阻擋一般簽到並提示前置條件不足，而不是回退成自由輸入。這樣能避免同一場會議同時出現「自動身份」與「手填身份」兩種資料來源，降低 ledger 去重、追溯與報表一致性的風險。

替代方案是抓不到身份時允許手動補填；但這會讓資料模型再次混入不可信的人工作業，與這次調整目的相衝突，因此不採用。

## Risks / Trade-offs

- [Risk] 前端原型暫時沒有真實登入上下文，實作時需要用 mock identity 補位。 → Mitigation：先在型別與 mock state 補出明確的 authenticated identity 來源，避免散落在 UI 判斷中。
- [Risk] 一般簽到與管理型流程分流後，彈窗狀態管理會比現在更複雜。 → Mitigation：把模式對應欄位需求整理成明確條件分支，避免用同一組 state 硬撐所有情境。
- [Trade-off] 抓不到身份時直接阻擋，短期看起來比手填更嚴格。 → Mitigation：在介面文案清楚指出需先完成登入/身份同步，並保留管理流程處理例外情境。
